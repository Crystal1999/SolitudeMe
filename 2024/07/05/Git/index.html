创建版本库

版本库又名仓库，英文名repository，可以简单理解成一个目录，这个目录里面的所有文件都可以被Git管理起来，每个文件的修改、删除，Git都能跟踪，以便任何时刻都可以追踪历史，或者在将来某个时刻可以“还原”。

创建一个版本库：

  • 找一个地方创建一个空目录（Windows注意别包含中文）
  • git init把这个目录变成git可以管理的仓库
    • 多了一个.git目录，默认是隐藏的，用来跟踪管理版本库的，不要动它

所有的版本控制系统，其实只能跟踪文本文件的改动，比如TXT文件，网页，所有的程序代码等等，Git也不例外。

版本控制系统可以告诉你每次的改动，比如在第5行加了一个单词“Linux”，在第8行删了一个单词“Windows”。而图片、视频这些二进制文件，虽然也能由版本控制系统管理，但没法跟踪文件的变化，只能把二进制文件每次改动串起来，也就是只知道图片从100KB改成了120KB，但到底改了啥，版本控制系统不知道，也没法知道。

Word格式是二进制格式，因此，版本控制系统是没法跟踪Word文件的改动

把一个文件放到git仓库：

  • git add readme.txt 把这个文件添加到仓库
  • git commit -m "update a file" 把文件提交到仓库，并附带本次提交的说明。

版本回退

一个几千行的文件每次都改了什么内容，命令可以告诉我们历史记录 git log显示从最近到最远的提交日志

$ git log

Author: Michael Liao 
Date:   Fri May 18 21:06:15 2018 +0800

    append GPL

commit e475afc93c209a690c39c13a46716e8fa000c366
Author: Michael Liao 
Date:   Fri May 18 21:03:36 2018 +0800

    add distributed

commit eaadf4e385e865d25c48e7ca9c8395c3f7dfaef0
Author: Michael Liao 
Date:   Fri May 18 20:59:18 2018 +0800

    wrote a readme file

简洁输出：git log --pretty=oneline

每提交一个新版本，实际上Git就会把它们自动串成一条时间线。

在Git中，用HEAD表示当前版本，也就是最新的提交。上一个版本就是HEAD^，上上一个版本就是HEAD^^。往上多了的话，写作HEAD~100

版本回退：git reset --hard HEAD^

关于--hard后面解释

再使用git log发现回退前的那个版本不见了。如果知道版本的commit id，就可以使用git reset --hard 1094a回到指定的版本。版本号没必要写全，前几位即可。

如果后悔了想恢复新版本但是找不到commit id怎么办？

git reflog记录了每一次命令，可以查到某次命令的commit id

工作区和暂存区

工作区：电脑里能看到的目录。

版本库：工作区有一个隐藏目录.git，这是git的版本库。版本库里存了很多东西，最重要的是stage（或者称index。暂存区）还有git自动创建的第一个分支master，一个一个指向master的指针HEAD

把文件往Git版本库里添加的时候，是分两步执行的：

第一步是用git add把文件添加进去，实际上就是把文件修改添加到暂存区；

第二步是用git commit提交更改，实际上就是把暂存区的所有内容提交到当前分支。

因为我们创建Git版本库时，Git自动为我们创建了唯一一个master分支，所以，现在，git commit就是往master分支上提交更改。

简单理解为，需要提交的文件修改通通放到暂存区，然后，一次性提交暂存区的所有修改。

一旦提交了又没有对工作区作任何修改，那么工作区就是“干净”的：

git status
On branch master
nothing to commit, working tree clean

管理修改

Git管理的是修改，当用git add命令后，在工作区的修改被放入暂存区，准备提交。但是，如果在工作区的修改并没有放入暂存区，由于git commit只负责把暂存区的修改提交，那么没有放入暂存区的修改不会被提交。

每次修改，如果不用git add到暂存区，那就不会加入到commit中。

撤销修改

git check out -- readme.txt 把这个文件在工作区的修改全部撤销，具体有两种情况：

  • readme.txt自修改后还没有被放到暂存区，那么撤销修改就回到和版本库一模一样的状态；
  • readme.txt已经添加到暂存区后，又作了修改，现在，撤销修改就回到添加到暂存区后的状态。

总之就是让文件回到最近一次git commit货git add的状态

git checkout -- file命令中的--很重要，没有--，就变成了“切换到另一个分支”的命令，在后面的分支管理中会再次遇到git checkout命令。

如果加到暂存区了，还没提交怎么撤销呢？

git reset HEAD readme.txt 撤销暂存区的修改，重新放回工作区。

git reset命令既可以回退版本，也可以把暂存区的修改回退到工作区。当我们用HEAD时，表示最新的版本。

删除文件

删除文件之后，工作区和版本库的状态不一致了，有两种情况：

  • 确实要从版本库中删除该文件 git rm 并且git commit文件就从版本库删除了
  • 删错了。从版本库中恢复 git checkout -- test.txt

没有被添加到版本库就被删除的文件，通过git无法恢复

远程仓库

本地git仓库和远程仓库的传输通过SSH加密。设置过程：

  • 创建SSH key。在用户主目录下，看看有没有.ssh目录，如果有，再看看这个目录下有没有id_rsa和id_rsa.pub这两个文件，如果已经有了，可直接跳到下一步。如果没有，打开Shell（Windows下打开Git Bash），创建SSH Key：

    ssh-keygen -t rsa -C "youremail@example.com"

    顺利的话，.ssh目录下产生两个文件id_rsa和id_rsa.pub，前者是私钥，不可泄露，后者是公钥可以公布。

  • 登陆github，添加ssh key，黏贴id_rsa.pub的内容。

原理：

当你尝试通过 SSH 访问远程 Git 仓库时，服务器会发送一个挑战（challenge）给你的客户端。客户端使用私钥对这个挑战进行加密，并将加密后的结果发送回服务器。服务器使用你之前上传的公钥对加密结果进行解密。如果解密成功，说明你拥有对应的私钥，身份验证通过。

添加远程库

加入先有了本地库，在远程库页面上，创建一个新的仓库，在本地库与远程库绑定：

git remote add origin git@github.com:XXX/XXX.git

在本地称呼远程的那个仓库的名字就是origin，相当于别名，这是Git默认的叫法，也可以改成别的，但是origin这个名字一看就知道是远程库。

本地库名字叫learngit，远程库全名叫git@github.com:michaelliao/learngit.git

但每次推送让你敲个全名你会疯的，所以起个别名origin。每次敲命令git push origin master的时候，git看到origin，就去当前库的配置文件里找，看看它的全名到底写的啥。你也可以起别的名字，如果有多个远程库，那必然不同的远程库对应不同的名字。

有一个朋友，名字叫做learngit ，他的手机号是 git@github.com:michaelliao/learngit.git 你在你的通讯录里把他的手机号备注为 origin,这样你以后给他打电话，发短信就是向 origin 发短信，打电话。origin 这个名字只在你的手机里有用

命令后面的地址加入写了别人的，没关系，相当于是本地与那个别人的仓库关联了，关联没有问题，但是，没有用，推送是推不上去的，因为别人那里没有添加你的SSH key公钥。

把本地库的所有内容推送到远程库上：

git push -u origin master

把本地库的内容推送到远程，用git push命令，实际上是把当前分支master推送到远程。（这个master是因为本地一开始默认的分支就是这个名字）

而且，由于远程库是空的，加上-u（实际上意思是upstream）设置了“上游分支”是 origin的master。因此，此时设置了本地（当前执行这个命令所在的分支）这个分支跟远程那个外号叫origin的仓库的master分支是对应关系。后面我们就可以简写git push和git pull了

删除远程库

如果添加的时候地址写错了，或者就是想删除远程库，可以用git remote rm 命令。使用前，建议先用git remote -v查看远程库信息。然后，根据名字删除。

此处的“删除”其实是【解除了本地和远程的绑定关系】，并不是物理上删除了远程库。远程库本身并没有任何改动。要真正删除远程库，需要登录到GitHub，在后台页面找到删除按钮再删除。

从远程库克隆

上面我们讲了先有本地库，后有远程库的时候，如何关联远程库。下面讲从远程库创建本地库的0~1

git clone git@github.com:michaelliao/gitskills.git

分支管理

创建与合并分支

一开始的时候，master分支是一条线，Git用master指向最新的提交，再用HEAD指向master，就能确定当前分支，以及当前分支的提交点

每次提交，master分支都会向前移动一步，这样，随着你不断提交，master分支的线也越来越长。

当我们创建新的分支，例如dev时，Git新建了一个指针叫dev，指向master相同的提交，再把HEAD指向dev，就表示当前分支在dev上。

Git创建一个分支很快，因为除了增加一个dev指针，改改HEAD的指向，工作区的文件都没有任何变化！

从现在开始，对工作区的修改和提交就是针对dev分支了，比如新提交一次后，dev指针往前移动一步，而master指针不变。

假如我们在dev上的工作完成了，就可以把dev合并到master上。Git怎么合并呢？最简单的方法，就是直接把master指向dev的当前提交，就完成了合并。

合并完分支后，甚至可以删除dev分支。删除dev分支就是把dev指针给删掉，删掉后，我们就剩下了一条master分支。

git checkout -b dev 其中的 -b参数表示创建并切换，相当于

git branch dev
git checkout dev

git merge dev操作需要切换到“目标分支”上运行该命令。也就是这个命令用于“合并指定分支到当前分支”

注意Fast-forward信息，Git告诉我们，这次合并是“快进模式”，也就是直接把master指向dev的当前提交，所以合并速度非常快。

当然，也不是每次合并都能Fast-forward

到切换分支使用git checkout ，而前面讲过的撤销修改则是git checkout -- <file>。容易混淆

实际上，切换分支这个动作，用switch更科学。

创建并切换到新的dev分支:

git switch -c dev

接切换到已有的master分支:

git switch master

解决冲突

比如：

git merge feature1
Auto-merging readme.txt
CONFLICT (content): Merge conflict in readme.txt
Automatic merge failed; fix conflicts and then commit the result.

使用git status查看冲突情况。同事冲突文件里的内容也会有变化，用<<<<<<<，>>>>>>>标记出不同分支的内容。修改之后再保存、合并。

用带参数的git log 也可以看到分支的合并情况。

git log --graph --pretty=oneline --abbrev-commit

分支管理策略

通常，合并分支时，如果可能，Git会用Fast forward模式，但这种模式下，删除分支后，会丢掉分支信息。【没有额外的合并提交（merge commit）。难以看出分支的具体合并点。】

如果要强制禁用Fast forward模式，Git就会在merge时生成一个新的commit，这样，从分支历史上就可以看出分支信息。

git merge --no-ff -m "merge with no-ff" dev

使用参数--no-ff表示禁用Fast forward。因为这次合并要创建一个新的commit，加上-m参数，把commit描述写进去。

自动合并模式，当待被合并的分支为当前分支的后代，那么会自动执行--ff方式是master指针直接快速移动到待合并的分支的位置。【实际上并没有发生真正的合并】当dev分支被删除后（比如开发时使用的feature分支），dev指针没了，只看得到master的干净线性的历史。

c4->dev 现在的位置，实际上是master的后代，master当初停在那里没动
c3
c2
c1->master, checkout -b dev
c0

非ff模式，在任何情况下都会创建新的commit进行多方合并。（即使被合并的分支为自己的直接后代）。

更一般的不能ff的、默认无法使用ff的情况：

d3->dev		.c4 ->master
d2			.c3
d1			.c2
c1->dev ....
c0
# 版本c0->c1
# c1 新建dev分支
# dev 产生了d1 d2 d3 commit
# master 产生了 c2 c3 c4 commit

master已经没办法通过移动指针来完成ff，不得不作出真正的合并。

  • 找到dev和master的公共祖先，找到c1,d3,c4三个节点的版本，如果有冲突需要处理
  • 创建新节点 e1，并且将三个版本的差异合并到e1，并且创建commit
  • 讲master和HEAD指针移动到e1

Bug分支

假如想创建一个分支来修复bug，但是当前正在dev上进行工作，还没提交。但是不想提交，工作只进行到一半，怎么办？

把当前工作现场“储藏”起来，等以后恢复现场后继续工作

git stash

然后git status查看工作区就是干净的，可以修复bug了。先确定在哪个分支上修复，比如是master，切换到master，然后git checkout -b issue-101修复完成后，切到master执行合并。

修复完成，回到原来那个工作，工作区是干净的。

git switch dev
git status
git stash list

恢复工作：

  • git stash apply恢复后，stash内容并不删除，需要用git stash drop删除
  • git stash pop恢复的同时把stash内容也删除了

再看stash list内容就看不到内容了。

可以多次stash，恢复的时候先查看，然后指定恢复某个：

git stash apply stash@{0}

在master分支上修复了bug后，dev分支是早期从master分支分出来的，所以，这个bug其实在当前dev分支上也存在。

要在dev上修复，我们只需要把修bug的那个提交所做的修改“复制”到dev分支。注意：只复制【这个提交】所做的修改，并不是把整个master分支merge过来。

git switch dev
git cherry-pick commit号

Git自动给dev分支做了一次提交，这次提交并不同于master的那个，因为这两个commit只是改动相同，但确实是两个不同的commit。这样我们就不需要在dev分支上手动再把修bug的过程重复一遍。

Feature 分支

每添加一个新功能，最好新建一个feature分支，在上面开发，完成后，合并，最后，删除该feature分支。

假如功能开发完，没合并，放弃了，要销毁。删除分支：

git branch -d branch-name

Git友情提醒，分支还没有被合并，如果删除，将丢失掉修改，如果要强行删除，需要使用大写的-D参数。

git branch -D branch-name

多人协作

当你从远程仓库克隆时，实际上Git自动把本地的master分支和远程的master分支对应起来了，并且，远程仓库的默认名称是origin。

现在可能是main不是master

查看远程仓库信息 git remote

或者看更详细的信息 git remote -v

如果git pull提示no tracking information，指定本地dev分支与远程origin/dev分支的链接

git branch --set-upstream-to=origin/dev dev

内存分析

jvm在执行时候,会在电脑内存条中,为自己这个应用程序申请一块内存空间,这块空间就是jvm的内存空间,这块内存空间很大,jvm对自己的这块内存空间内部进行了一些划分,分为不同区域,已完成不同功能,jvm的内存的分区: 栈内存, 堆内存, 方法区, 本地方法区, 程序计数器;一共五块;

jvm内存分块的描述:

  1. 栈内存[常用]: 用于执行方法的区域,当有多个方法在执行时,会为每个方法划分一块空间, 每个方法所占空间称之为”栈帧”, 每个方法进入栈内存的过程称之为”进栈”, 栈内存管理数据的特点: 先进后出

  2. 堆内存[常用]:用于存储数组,对象等数据比较大的数据, 这些数据一般都是引用数据类型

  3. 方法区[常用]: 用于存储字节码文件(.class), 常量, 存储静态变量

  4. 本地方法区: 用于存储执行的方法,这些本法都是使用C和C++写的

  5. 程序计数器: 用来控制程序的执行流程,决定要执行程序中哪行码

数组名赋值

数组名赋值给另一个数组名, 此时两个数组名存储一样地址, 他们两指向同一块堆区空间

因为数组名中存储的是地址, 所以数组名相当于指向堆区的某块内存空间, 这种现象称之为内存的引用; 所以数组就是一种引用类型数据;

所以以后咱们说的引用数据类型变量,就指的是变量中存储了某个内存地址

  1. 第一次使用某个类型（类）新建对象的时候,会把该类的字节码文件加载到方法区,第二次的时候就不再加载

  2. 在堆区创建的对象, 内部空间中会持有一份该类字节码文件在方法区的地址

  3. 每个对象在调用自己的对象方法时,该方法中都会有一个地址指向这个对象在堆区的地址

  4. 每使用一次new, 就会创建出一个新的对象, 都会在堆区申请新的空间

成员变量与局部变量

局部变量: 属于方法,方法运行时会入栈,所以局部变量就在栈区

成员变量: 属于对象,对象的位置在堆区, 所以成员变量在堆区

局部变量: 没有默认初始化的值, 不初始化不能使用;

成员变量: 有初始化的值, 系统会自动申请空间并初始化

​ 引用类型初始化值: null

​ 基本数据类型: 整型: 0

​ 布尔: false

​ 小数: 0.0

​ 字符 ‘\u0000’

局部变量:随着方法的调用被创建出来, 方法调用结束就消失

成员变量:随着对象的创建而存在,随着对象的消失而消失. 即使栈内存的对象名被销毁了,对象也不一定立即被销毁.这个对象只能通过垃圾回收机制,才能被销毁;

关于“栈内存的对象名”的理解：

比如，有一个方法，里面new了一个某类的对象。然后这个方法结束的时候，这个对象的地址指针没了，但是呢，这个对象里面的 int age什么的属性，占得内存在堆空间里面，还没释放，那一块是靠垃圾回收机制自己回收的

创建对象的步骤：

  2. 在栈内存中创建一个该对象引用(对象名),用来存储对象在堆区的地址

  3. 在堆区开辟内存空间,给每个成员变量分配空间

  4. 默认初始化先给成员变量赋值

  5. 显示初始化再给成员变量赋值

  6. 通过构造方法给成员变量赋值

  7. 最后把堆区对象的地址赋值给栈内存中对象名

静态static

静态内容加载时机:

  • 随着类的.class文件的加载而加载;

  • 静态的变量的随着类被加载到方法区,直接会给开辟一个静态区区域存储这个静态变量;

一个类的.class文件什么情况下会被加载到方法区:

  1. 使用这个类创建对象
  2. 使用这个类访问静态的内容(静态属性和静态方法)
  3. 使用反射的方式操作了这个类

静态方法既可以使用对象名访问也可以使用类名访问

静态方法中不能访问非静态的内容(成员变量和对象方法)

​ 原因:静态方法可以在没有创建对象之前就已经加载完成; 非静态(对象)的内容只有在创建对象之后才会完成加载;如果静态方法中能够访问非静态的变量,那么就相当于在对象没有创建之前,就访问了对象的数据,这明显不合理;

非静态的方法可以访问静态的内容:

​ 因为静态的内容是早于对象创建之前就完成了,所以对象中能够访问静态的内容;

静态方法中不能使用this关键字

​ 原因: this代表是本类创建出来的对象, 而静态方法可以在对象没有创建之间就可以调用;如果能使用this关键字, 相当于在创建对象之前访问了this, 这明显也不合理;

         数组的遍历
         @param arr  要遍历的数组名, 不能为null
     */
     public static void printArray(int[] arr){
        for(int i = 0; i < arr.length; i++){
            System.out.print(arr[i]+"  ");
        }
        System.out.println();
     }
     /**
        获取数组中的最大值
        @param arr 要操作的数组名, 不能为null
        @return 获取的数组的最大元素
     */
     public static int getMax(int[] arr){
         int max = arr[0];
         for(int i = 0; i < arr.length; i++){
            if(max < arr[i]){
                max = arr[i];
            }
         }
         return max;
     }
     /**
        获取数组中的最小值
        @param arr 要操作的数组名, 不能为null
        @return 获取的数组的最小元素
     */
     public static int getMin(int[] arr){
         int min = arr[0];
         for(int i = 0; i < arr.length; i++){
            if(min > arr[i]){
                min = arr[i];
            }
         }
         return min;
     }

  
     /**
        数组的元素的交互
        @param arr 要操作的数组名, 不能为null
        @param i 要交换元素的下标
        @param j 要交换元素的下标
     */
     public static void swap(int[] arr, int i, int j){
            int temp = arr[i];
            arr[i] = arr[j];
            arr[j] = temp;
     }
     
     /**
        数组的反转
        @param arr 要操作的数组名, 不能为null
     */
     public static void inverse(int[] arr){

                swap(arr, i, j);
        }
     }
}

继承

定义一个A类, 定义B类, 我让A类继承自B类,A类中就可以拥有B类中所有内容;

父类和子类的关系:

父类: 就是被继承类, 别称: 超类, 基类

子类: 就是用于继承的类. 别称:派生类

好处:

  1. 提高了代码的复用性

  2. 提高了代码的可维护性

  3. 是类中多态实现前提

弊端:

  1. 提高了代码与代码之间耦合性

  2. 耦合性:事物与事物之间的关系,就是耦合性, 也就事物与事物之间互相影响的程度

开发的原则: 面向对象最终的目标要实现: 高内聚:(一个事物自己可以完成所有的功能), 低耦合(对于其他代码的依赖性非常低)

父类中私有成员变量不能继承(不能在子类中直接访问)

其实在子类对象中,仍然包含了父类中私有成员变量

只不过在子类中,不能直接去使用这些私有的内容

父类中的构造方法不能继承

继承中成员变量的关系：

  1. 在子类中定义和父类中不同名成员变量, 在子类中,既可以使用子类访问自己的成员变量,也可以访问父类中成员变量;

  2. 在子类中定义和父类中同名的成员变量, 子类会根据就近原则,只能访问到自己定义的成员变量

    在子类中方法中：如果要访问某个变量名, 优先在自己方法中寻找,有没有定义这样一个局部变量名, 如果找到就直接使用；如果没找到就去自己类中查看没有定义这个变量名的成员变量,如果找到就直接使用；如果没找到就去父类中寻找,父类中有没有定义成员变量名,如果找到就直接使用；如果没有找到,继续向父类的父类中寻找, ….一直找到java中所有类的根类,Object这个类,如果有就使用;如果,没有就会编译报错;

继承中构造方法的关系：

  1. 在初始化子类数据之前,必须先完成对父类数据的初始化(因为在初始化子类的时候,可能会用到父类的数据, 所以必须先把父类的内容初始化完成)

  2. 如果在子类构造方法,没有显示的调用任何其他的构造方法, 在构造的第一句,系统会默认给你加上一句代码: super(); 默认访问父类空参的构造方法;

  3. 如果子类在自己的构造方法,手动的加上父类构造方法的调用, 那么系统就不会给你添加任何构造方法的调用;

  4. 自己可以调用自己的构造方法,访问自己的其他构造方法;此时也不会添加super();

  5. 构造方法不能自己调用 （指的是像递归似的调用自己）

  6. 构造方法的调用,也不能写成循环调用（指的是不能：A（int i）构造方法里调用 A（String s），后者里面又调用了前者，形成循环调用）

  7. ★在一个构造方法中, this语句和super语句只能出现在这个构造方法的第一句;

  8. ★在一个构造方法中,this语句和super语句两者只能出现一个;

  9. 总结:

  • 子类在调用自己构造方法时,默认调用父类的构造方法

  • 子类中如果访问其他构造方法, 必须把它this语句和super语句写在第一句; ⑦

  • 子类的构造方法中,this语句和super语句两者只能出现一个

继承中成员方法的关系：

  1. 在父、子类中,如果写的都是不同名的方法

    子类既可以访问父类的成员方法,也可以访问自己的成员方法

  2. 在父、子类中,出现同名的方法

    在父、子类中,出现了一模一样的方法,但是两个方法内部实现不一样。在子类调用这个方法的时候,只会走子类对这个方法的实现。（相当于覆盖了，也即重写）

  3. 重载: 在 同一个类 中,方法名一致,参数列表不同, 与返回值类型无关 （注意 是参数列表不同！！！）

    重写: 在父子类中,方法名一致, 参数列表一致,返回值类型一致

    java中可以通过@Override检测一个方法是不是子类重写父类的方法

  4. 方法重写注意事项:

  • 重写方法必须是父类定义的方法;

  • 重写方法可以保存父类的实现,保存的方式super.重写的方法名(实参),也可以不要父类的实现;

  • 重写的方法,访问权限不能比父类小

  • 父类的方法如果用private修饰, 子类是无法重写

多态

java中多态指的是,父类类型的引用可以指向子类对象。本质上是同一个事物有不同的名称和描述。

多态实现的前提

  1. 要有父子类关系, 没有继承就没有多态

  2. 方法的重写

  3. 父类类型引用指向子类对象

多态中成员变量的访问特点：

  1. ★编译的时候看左边, 运行的时候也看左边;

  2. 编译的时候,要看[=]左边的引用类型中,是否定义了该变量名,如果没有,编译就报错;如果有,编译就成功;

  3. 运行的时候,要看[=]左边的引用类型,左边引用类型中如何的赋值的,取出的就是左边引用类中的值;

多态中成员方法的访问特点：

  1. ★编译看左边,运行看右边

  2. 编译的时候, 要看[=]左边的引用类型中,是否定义了该方法,如果有,就编译通过;如果没有,就编译失败;

  3. 运行的时候,要看[=]右边,对象所属的类中,如果对这个方法进行实现,最终运行的就是子类对这个方法的实现;

多态中静态方法的访问特点：

  1. ★编译时看左,运行时看左

  2. 编译的时候,要看[=]左边的引用类型中,是否有该静态方法名,如果有就编译通过,没有编译报错;

  3. 运行的时候,要看[=]左边的引用类型中,如果对该静态方法进行的实现,调用父类中对该方法的实现

静态方法不能被重写！

静态的方法可以被继承，但是不能重写。如果父类中有一个静态的方法，子类也有一个与其方法名，参数类型，参数个数都一样的方法，并且也有static关键字修饰，那么该子类的方法会把原来继承过来的父类的方法隐藏，而不是重写。

通俗的讲就是父类的方法和子类的方法是两个没有关系的方法，具体调用哪一个方法是看是哪个对象的引用；这种父子类方法也不在存在多态的性质。

向上转型

  • 使用子类的引用指向指向的子类的对象(正常情况)

  • 多态中,使用的是父类的引用指向子类的对象(向上转型)

  • 本质:缩小了对象本身的访问范围, 减少了它的访问权限,只能访问父类中的内容

向下转型:

  • 概念: 让一个指向子类对象的父类引用,恢复成子类的引用

  • 格式: 子类类型 引用名称 = (子类类型)父类类型引用;

  • 本质:恢复子类对象的访问权限,访问范围

抽象类

可以定义抽象方法的类,就是抽象类

拥有抽象类的好处, 避免了父类对某个方法提供实现, 但是它所有子类们继承过去这个都要重写,那么父类就没必要提供这个方法的实现了,直接把这个方法定义为抽象方法即可

没有方法实现部分, 连大括号都不要写, 在最后面添加一个分号,表示方法定义结束。public abstract void eat();

总结：

  • 抽象类中可以有抽象方法,也可以没有抽象方法;抽象方法必须存于抽象类中
  • 抽象类不能实例化。可以定义抽象类子类,由子类去创建对象, 调用方法
  • 如果抽象类的子类完全实现了抽象类中抽象方法,难么这个子类就是一个普通子类,就可以创建对象。否则这个子类还是一个抽象类, 必须使用abstract, 同样无法实例化
  • 抽象类可以定义成员变量, 也可以定义常量。但是由于不能实例化，所以用不了。
  • 抽象类可以有构造方法。
  • 抽象类中可以有非抽象的成员方法;这里定义方法内容一般子类不会去重写

接口

接口 interface 的本质:就是规则

接口本质还是类, 发现接口编译之后生成的是.class文件

接口定义者只需要说明应该有哪些方法,不需要管具体谁来实现这些方法, 实现了 规则的定义和规则实现的分离

接口中只能定义抽象方法

实现接口的格式：

class 类名 implements 接口名 {

重写接口定义的所有抽象方法;

}

接口的实现类,可以看做是接口的子类, 可以使用接口类型引用指向实现类对象

一个类如果实现了接口中所有的抽象方法,那么它就是一个普通类。否则它必须是一个抽象类

接口中不能定义成员变量, 定义的属性都是静态常量属性

定义属性是默认前面会加: public static final 。建议手动加上这些修饰符

接口不能定义构造方法, 因为接口中就没有成员变量, 就没有定义构造方法的意义

接口中不能有成员方法，因为接口中只能定义抽象方法;

定义抽象方法是: 可以只写返回值类型 方法名(参数列表): 默认会在前面加上 public abstract。建议加上

类和接口的关系：实现 implements

可以单实现，可以多实现，不能多层实现。（一旦实现接口就必须是一个 类）

注意：”不能多层实现”理解为：

不是不能：interface –implements–>abstract class -> class

而是不能：interface –implements–> interface

★多继承: 接口1 extends 接口2, 接口3, 接口4 { }

多层继承: 接口1 extends 接口2{ } 接口2 extends 接口3{}

内部类

定义在一个类内部的类,就是内部类,内部类可以出现的位置: 类体中, 方法体

内部类

  • 成员内部类
    • 普通成员内部类
    • 私有成员内部类
    • 静态成员内部类
  • 局部内部类
    • 定义方法中类

根据有无名字，分为有名的、匿名的

普通成员内部类：

定义格式:

class 外部类名 {
   class 内部类名 {
        成员变量;
        成员方法;
    }
}

说明

  1. 普通成员内部类,可以在它的外部类中直接使用

  2. 如果出了外部类,普通成员内部类的使用方式:

    外部类名.内部类名 内部类对象名 = new 外部类().new 内部类();

  3. 内部类可以直接使用它所在外部类中成员变量和成员方法

私有成员内部类

也是一个成员内部类,只是在普通成员内部类前加一个private修饰符

注意事项:

  1. 可以在外部类中直接创建内部类对象, 不能直接使用内部的属性和方法

  2. 不能在外部类以外直接创建内部类对象

    也即不能：

    外部类名.内部类名 内部类对象名 = new 外部类().new 内部类();

  3. 私有成员内部类可以直接使用它所在外部类中成员变量和成员方法

静态成员内部类：

也是一个成员内部类,只是在普通成员内部类前加一个static修饰符

注意事项:

  1. 静态内部类只能使用外部类静态的属性和方法, 对象的内容不可以访问

  2. 外部类可以直接使用静态成员内部类

  3. 外部类之外也能访问静态成员内部类

​ 访问的格式: 外部类名.内部类名 对象名 = new 外部类名.内部类名();

局部内部类：

定义在方法中的类

注意事项：

  1. 只能它所在的方法范围里,才可以使用

  2. 出了方法就不能使用这个局部内部了

  3. 只有调用局部内部类所在的方法,才会创建局部内部类对象

匿名内部类：

本质: 快速的为一个类创建一个子类,或者快速为一个接口提供一个实现类

临时的、方便的

匿名内部类存在的前提：继承自一个类或者实现某个接口

new 父类名(){
    需要重写的方法;
};

new 接口名(){
    重写接口中抽象方法
};

总结:

  1. 如果匿名内部类创建的对象,没有使用变量接收, 只能使用一次;

  2. 如果想多次使用匿名内部类对象,使用父类或者接口类型变量存储匿名内部类对象

package com.OOP;
public class Person {
    private String name;
    private int age;
    public Person() {
        super();
        // TODO Auto-generated constructor stub
    }
    public Person(String name, int age) {
        super();
        this.name = name;
        this.age = age;
    }
    public String getName() {
        return name;
    }
    public void setName(String name) {
        this.name = name;
    }
    public int getAge() {
        return age;
    }
    public void setAge(int age) {
        this.age = age;
    }
    @Override
    public int hashCode() {
        // null  0
        final int prime = 31;
        int result = 1;
        result = prime * result + age; // 31 * 1 + 0
        //  31 * 31 + 0
        //  961
        result = prime * result + ((name == null) ? 0 : name.hashCode());
        return result;
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj)
            return true;
        if (obj == null)
            return false;
        if (getClass() != obj.getClass())
            return false;
        Person other = (Person) obj;
        if (age != other.age)
            return false;
        if (name == null) {
            if (other.name != null)
                return false;
        } else if (!name.equals(other.name))
            return false;
        return true;
    }
    @Override
    public String toString() {
        return "Person [name=" + name + ", age=" + age + "]";
    }
}

== 和 equal

比较的内容不同

== 可以比较任意数据类型的数据, 即可以比较基本数据类型,也可以比较引用数据类型

equals只能比较引用类型数据,不能用来比较基本数据类型

比较的规则不同

​ == 如果是基本数据类型,比较数值, 如果是引用数据类型比较的是地址;

​ equals 重写该方法之前,比较的是地址; 重写之后,比较的是属性值;

异常

异常就是一个对象.描述那些不符合正常情况信息,异常的内容包含: 异常的原因,异常的位置,以及异常的类型

我们可以对异常进行捕获,跳转或者结束程序

出现了错误必须修改源代码, 出现了异常可以捕获

Throwable 是异常体系的顶层父类,其他的异常或者错误都是Throwable子类。Error 错误,是Throwable的一个子类,用于描述那些无法捕获和处理的错误信息,属于非常严重的问题。

Exception: 异常,也是Throwable的子类, 用于描述那些可以捕获和处理的错误信息, 属于不太严重的问题。

RuntimeException: 运行时异常,是Exception的一个子类, 比较特殊,这种异常在编译的时候不会检测,编译能够正常通过,只有出现对应问题时这种异常才会被抛出。

如 错误的强制类型转换、数组访问越界、访问null指针

如果出现RuntimeException，就一定是你的问题 （指程序员）

JVM中默认的异常处理机制：

某个方法执行的过程中，如果调用其他方法产生了异常,在其他方法执行时候如果自己没有手动处理异常，就会把这个问题抛出方法的调用者中，调用者也没有处理异常，一层一层向上抛出，最后到达JVM，JVM把这个异常抛出，以红色的信息显示异常，同时把程序结束。

  1. 声明式处理。最简单的方式。方法就是在调用有编译异常的方法名后使用throws关键字 将对应编译异常抛出即可

throw 关键字

与 throws 不同的是，throw 语句用来【直接拋出一个异常】

运行时异常，调用者可以选择处理也可以不处理。

throw 语句，要么用try catch包着来捕获，要么不捕获，但是要声明throws。

throws关键字

当一个方法产生一个【它不处理】的异常时，那么就需要在该方法的头部【声明】这个异常，以便将该异常【传递到方法的外部】进行处理。使用 throws 声明的方法表示【此方法不处理异常】。

使用 throws 声明抛出异常的思路是，当前方法不知道如何处理这种类型的异常，该异常应该由向【上一级的调用者】处理

如果有多个异常类，它们之间用逗号分隔。可以直接使用Exception替换掉

如果声明的异常含有父子类关系，不会报语法错误

throws所在的方法被上层谁调用了，那里要么用try catch包着来捕获，要么不捕获，但是要继续声明throws。

使用 throws 声明抛出异常时有一个限制，是【方法重写】中的一条规则：

子类方法声明抛出的异常类型应该是父类方法声明抛出的异常类型的【子类或相同】，子类方法声明抛出的异常【不允许多于】父类方法声明抛出的异常。

throws 关键字和 throw 关键字在使用上的几点区别如下：

  • throws 用来【声明】一个方法可能抛出的所有异常信息，表示出现异常的一种【可能性】，但【并不一定会发生】这些异常；throw 则是指拋出的一个具体的异常类型，执行 throw 则【一定】抛出了某种异常对象。

  • 通常在一个方法（类）的声明处通过 throws 声明方法（类）【可能】拋出的异常信息，而在方法（类）内部通过 throw 声明一个具体的异常信息。

高速缓冲流

  1. BufferedInpuStream和BufferedOutputStream
  2. 这两个本身不具备读写的能力,只是在某个具体的流对象基础上,对其进行了加强的操作, 例如可以加强: FileInpuStream或者 FileOutputStream. 这两流原本的效率比较低, 通过缓冲流加强之后, 效率就会得到升高
  3. 缓冲流构造方法

​ BufferOutputStream(OutputStream os):将要指定输出流对象传入构造方法，已达到对其增强目的,返回值是BufferOutputStream对象

  1. 增强原理

​ BufferInputStream高效的原理: 该类在内部准备一个数组, 用来存储读取的字节信息,当外界调用一次read()想要获取一个字节信息, 但是该对象从文件一次读取8192(8192=1024*8)字节内容存储到他的缓冲数组中,你调用read方法时只给你返回一个. 当你再次调用read方法,就不要需要去磁盘中读取字节了,直接从缓冲数组中获取一个字节,由于读取数据是从内存中获取的, 所以效率非常快, 当8192字节全部从缓冲数组中读取完之后, 在调用read方法, 会再次一次读取8192字节放到缓冲数组中, 并返回值一个, 每次都是这样操作, 这样可以减少IO次数,从而达到效率提高

​ BufferOutputStream高效原理: 该类在内部也准备一个数组, 用来存储你每次要写出去信息, 当你调用write方法向往磁盘上写出一个字节, 不是直接写到磁盘上,而是写到缓冲数组中,缓冲数组是8192字节,再次调用write还是写到缓冲数组,直到这个缓冲数组被写满,才会把数据写到磁盘上, 这样做写了8192次, 才会真正操作一次磁盘, IO次数减少,效率就提高

关于close 与 flush

  1. close方法被调用前会自动flush方法

  2. close方法的作用关闭流对象, 一旦调用close方法, 流对象就不能接着使用了

  3. flush方法被调用后,仅仅是把缓冲区中数据【刷新】到目标文件,而不是把流对象关闭, 也就是说调用完flush方法可以接着使用流对象

  4. flush【不要频繁的调用】, 调用多了就会使缓冲区的作用失效,增加IO时间,降低IO效率

            fis.close();
            //不空尝试关闭 并保持警惕出现异常
        } catch (IOException e) {

        }
    }

多线程

多线程的实现方式

第一种: 继承

Thread类, 这个类是线程类, 可以使用它创建一个线程。这个线程中已经把任务给写死了,无法添加开启线程先执行任务.

所以我们要定义一个类继承Thread，重写它提供和线程任务相关的方法

步骤:

  1. 定义一个类, 继承自Thread

  2. 重写自定义类中run方法,用于添加自己的线程任务

  3. 创建自定义类的对象,也是一个线程对象

  4. 调用线程开启的方法: start(), 调用开启线程方法后,此线程会【自动调用】run方法。【run里面是那个线程的执行逻辑】

不要使用自定义的线程类创建出来的对象调用run方法。因为此时没有开启线程，仅仅自己【调用一次】自己方法，开启线程要使用start()方法，开启线程之后,线程会自动调用run方法

第二种：实现接口

Runnable是一个任务接口, 只需要定义一个类实现Runable接口, 创建一个任务对象,将这个任务对象绑定到某个线程对象,也能实现多线程

步骤:

  1. 定义一个任务类, 实现Runable接口

  2. 重写接口中run方法,添加自定义的任务

  3. 创建一个任务对象, 表示一个任务

  4. 创建一个Thread对象, 将任务对象绑定到线程对象上

  5. 调用线程对象start方法, 开启多线程

// 创建任务对象
MyRunable run = new MyRunable();
// 将任务对象绑定到线程上
Thread t = new Thread(run);
// 开启多线程
t.start();
Thread t2 = new Thread(run);
t2.start();
Thread t3 = new Thread(run);
t3.start();
//"同一个任务可以被多个线程执行"

两种方式的比较：

继承式 实现接口式
原理 调用start方法, 内部调用start0方法,start0是本地方法(native), 由于虚拟机的实现是C语言实现的方法, 所在我们在java中看不到start0的内部实现,但是在start0会反过来调用java中run方法,由于run方法子类已经重写了,所以最终执行时子类的实现 Thread构造方法中,将Runable的实现类对象传入的构造方法, 通过这个构造方法,就把任务对象赋值给了Thread的target属性;当调用start方法时, 内部调用start0方法, start0会自己方法中反过来调用java中run方法.由于target属性已经被赋值了, 在run方法,使用target调用【target自己的run方法】,从而实现多线程
设计模式 某个类继承Thread类型, 那么就无法继承其他类了,就会限制我们对这个类设计,这个扩展性就会比较差 某个类通过实现Runable接口,完成多线程的设计, 这个类还可以继承其他的类,这个类扩展性就会比较强
灵活性 将线程对象和任务绑定死了,不能分离, 耦合性比较高,灵活性比较差 将线程对象和任务对象分离,耦合性比较低,灵活性比较强. 同一个任务可以被多个线程执行, 某个线程对象可以执行其他的任务对象。将来还可以将任务对象，提交到线程池中，由线程池中某个线程来执行任务。 任务被多个线程对象执行，方便线程之间进行数据交流。

使用同步代码块之后效果:

可以通过java提供一个类 State，是Thread类的一个内部类，获取到线程中每一个精确的状态。这个类封装的线程的状态和理论中的状态不完全匹配

Thread.State getState() 通过这个方法获取到线程每一状态,返回值就是内部类对象。可以直接print（state对象）

返回值有哪些:

NEW: 新建态, 还没有开启线程

RUNNABLE: 在jvm中运行的线程转台, 运行态

BLOCKED: 阻塞态(死锁的时候, I\O)

WATING: 阻塞态(锁对象调用wait方法,等待被notify唤醒的过程)

TIMED_WAITING: 阻塞态(调用sleep方法的时候会进入这种状态)

TERMINATE: 线程的死亡状态

线程池

如果在系统,有大量任务,要创建大量的线程对象,线程对象的创建和销毁很耗时,尤其对于任务量比较小的线程,可能执行任务所花时间比创建和销毁时间还短, 浪费就会特别大

有线程池之后：

在没有任务的时候,先把一些线程对象【提前准备好】,存储到一个容器中, 当一旦有了任务,就不需要去创建线程对象了,直接去容器中获取一个线程对象,执行这个任务

任务执行完的时候,不是把线程对象销毁,而是把线程对象【重新放回】线程池中

缺点: 如果一直没有任务执行, 总会有一部分线程对象占用资源.

步骤

  1. 获取线程池对象

    通过Executor工具类,获取线程池对象: 方法名为: newFixedThreadPool(int nThreads)

    通过nTreads参数执行获取线程池中,初始化几条线程对象

    ExectorService pool = Executor.newFixedThreadPool();

  2. 准备任务对象

    创建任务对象 实现Runnable接口自定任务

  3. 将任务提交到线程池中

    submite(Runnable run)

线程池工作的流程

将一个任务对象提交到线程池中, 线程池就会去查询有没有空闲线程,如果有就把这个空闲线程取出来,用来执行这个任务。如果没有空闲的线程,就需要等待用完归还的线程

线程池对象的其他方法:

shutdown(): 可以结束线程, 会把已经提交的任务都执行完,结束之前没有提交的任务就不在执行了.shutdown之后还提交任务就会抛出异常.

shutdownNow(): 结束线程池, 已经开始的任务保证执行完,但是已经提交还没有的任务就不再执行了.返回等待任务列表

public class SingletonLazy {

    private static SingletonLazy single;

    public static SingletonLazy getInstance() {
        // 外能判断不能省略
        // 省略了每次都要获取锁对象,还要释放锁对象,效率就会比较低
        if(single == null) {
            // 同步代码块,为了保证多线程下对象创建唯一性
            synchronized (SingletonLazy.class) {
                //假设没有第二层检查，
                //那么第一个线程创建完对象释放锁后，
                //后面进入对象也会创建对象，会产生多个对象
                if(single == null) {
                    single = new SingletonLazy();
                    System.out.println("单例对象被创建了");
                }
            }
        }
        return single;
    }
}

三要素

三要素指的就是要进行网络编程必备的条件: IP, 端口号, 网络协议

几个特殊的IP需要记忆:

端口号作用要来区分一个进程的,通过端口号可以快速找到此电脑上某一个进程.

使用两个字节来表示端口号: 0~65535

TCP/IP协议簇, 俗称的网络四层模型, 每一层都有每一层的协议, 每一层的协议完成工作都是不一样的.

应用层: http,https, FTP

传输层: UDP, TCP

网络层:IP协议, 路由协议

物理层: 底层硬件设备,网线.保证数据传输完整性

UDP协议中两端分别为: 发送端和接收端

TCP协议中两端分别为: 服务端和客户端

UDP和TCP协议区别:

UDP协议是一个面向【无连接】协议(发送数据的时候不需要确定接收端已经在等待了). UDP的使用场景: 发短信, 广播. 生活中的比如: 发快递, UDP协议使用的时候发出的消息【不需要确认接收端是否接的到】, 所以他【不安全】, 【效率高】. 使用UDP协议时区分发送端和接收端, 不区分服务端和客户端. 接收端如果要想接收消息必须提前执行, 发送端后执行.

TCP协议是一个【面向连接】的协议(发送数据时必须确认两端已经联通,才能把数据发送出去).TCP使用场景: 打电话 发送数据一端【必须确定接收端已经存在】, 才会把数据发送出去. 这样就会【提高安全性】, 【效率比较低】. TCP严格区分服务端和客户端. TCP协议往往基于请求的协议. 客户端必须主动找服务端, 服务端才会给你返回消息

概述

在程序的运行期间,对于任意一个类, 都能知道这个类【所有的】属性和方法(公开私有的都可以获取到),且可以通过反射机制【调用】该类中所有的方法,【访问所有属性】

所有的类的字节码文件加载到方法区之后, 都会把个【类的字节码文件编译为一个对象】,这个对象就是【类对象】,所有的类对象都是【Class类型】的

Class类的 对象

Class表示一个正在运行期间类和接口生成的对象, 基本数据类型也有对应的类对象

类对象只能获取不能创建

Class对象获取方式:

  1. 通过Object类中getClass()方法

  2. 通过类名.class也能获取该类的类对象

  3. 通过Class类的forName(String name)方法,获取一个类对象, name一定是这个类全类名(包名+类名)【反射的方式】

前两种获取类对象时必须要有明确的数据类型的存在

lambda表达式

Lambda表达式:就是对一个接口匿名内部实现的一种简化

java8之后引入一个新的操作符, -> 这个符号称之为”箭头符号”, Lambda运算符

Lambda表达式格式说明:

  • 必须有箭头运算符 “->”

  • 箭头左边: Lambda表达式的参数列表(就是你定义在接口的抽象方法中参数部分, 不需要写参数类型,只写参数名) 参数名只要符合标识符命名规则即可

  • 箭头右边:表示Lambda表达式的实现体. 就相当于方法的方法体. Lambda体

语法格式1: 没有参数,没有返回值抽象方法

​ ()->System.out.println(“Hello Lambda”);

​ 注意:没有参数小括号不能省略

语法格式2: 有一个参数,没有返回值

​ x->System.out.println(“你好 ”+x);

​ 注意: 如果只有一个参数, 小括号可以省略

语法格式3: 有多个参数, 没有返回值

(x, y)->System.out.println(x+y);

语法格式4: 有多个参数, 方法体中多条语句, 不能省略大括号

​ (x, y)->{double pow = Math.pow(x, y); return pow};

注意:如果Lambda表示只有一条语句,大括号可以省略不写;

但是方法体中只有一个return语句,如果想省略大括号,必须把return去掉, 否则不能省略大括号

Inter5 i3 = (x, y)->{
 return Math.pow(x, y);
};

Inter5 i4 = (x, y)-> Math.pow(x, y);

函数式接口

Lambda表达式使用前提, 接口必须是一个函数式接口

函数式接口的定义:

​ 在一个接口中,只有一个抽象方法,那么这个接口就是一个函数式接口

函数式接口的校验方式: 使用注解: @FunctionInterface

​ 一个接口使用此注解修饰之后，如果是函数式接口，编译就能通过，否则编译报错

函数式接口作用:

​ 函数: 具有待定的功能代码段。函数不是java的内容,java中叫方法. 函数是从C语言中舶来的概念。

​ 函数式接口作用，函数式接口中只有一个抽象方法，此抽象方法也就只属于这个接口，也就意味着访问这个接口就是在访问此抽象方法，访问此抽象方法也就是在访问此接口。意思就是把一个抽象方法和接口就绑定到一起了。

​ Java不支持函数作为方法的参数传递,有了函数式接口就可以间接的解决这个问题, 当你定义一个方法,参数类型是函数式接口类型的. 调用此方法时就可以创建一个此函数式接口实现类对象进去.此实现类对象只有一个方法, 也就【意味着把这个方法当做参数传递】了.

相当于有一个接口（抽象类），我实现了该接口，新建了一个对象，且重写（实现）了方法，将这个对象传进去。一并带进去的还有我实现的方法。【看代码案例】

方法引用

Lambda表达式就是函数式接口一个实现对象, 此Lambda必须要提供抽象方法的实现方式, 如果发现此实现方法【已经被其他类或者其他对象实现了】, 就不需要自己再写Lambda体了,就可以直接引用其他类或者对象对这个方法的实现

格式:

函数式接口 名称 = 对象名 :: 方法名
函数式接口 名称 = 类名 :: 静态方法名

方法引用作用

把已经实现的方法, 作为一个数据, 赋值给函数式接口的引用.

而且这个引用既可以当做方法的参数也可以当做方法的返回值

方法引用本质

其实任意一个方法,都是一个函数式接口对象.

任意一个方法都可以通过方法引用替换掉一个Lambda表达式, Lambda表达式就是函数式接口的实现对象, 方法也就是函数式接口的实现对象

public class FunctionRef {

  1. filter()根据其内的判定规则筛选出true的元素
